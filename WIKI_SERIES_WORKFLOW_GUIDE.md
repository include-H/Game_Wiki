# 游戏 Wiki 系列编写工作流指南

> 说明：本文不是单作写作模板，而是一份面向当前仓库使用场景的“执行工作流说明”。它总结的是一次完整系列 Wiki 编写任务应如何落地，包括前期准备、资料核对、分批写作、目录同步、质量自检与防止上下文污染的方法。可与 [WIKI_WRITING_GUIDE.md](./WIKI_WRITING_GUIDE.md) 搭配使用。

---

## 1. 这份文档解决什么问题

`WIKI_WRITING_GUIDE.md` 主要解决“条目应该怎么写”的问题，而这份文档主要解决“任务应该怎么做”的问题。

换句话说：

- 前者偏写法
- 后者偏流程

如果后续要持续扩写仓库里的系列 Wiki，最稳妥的做法不是一上来就埋头写正文，而是先把工作流程固定下来。这样才能尽量减少下面这些常见问题：

- 把不同系列的信息混在一起
- 把前一作的版本关系套到后一作
- 先写了几篇单作，结果系列目录没有立起来
- 单作写完后忘记同步总目录
- 官方资料、商店页、Wiki 和中文常见说法互相打架时，没有统一判断顺序
- 一次写太多，导致事实核对开始失真

## 2. 总体原则

当前项目最推荐的执行原则有四条：

### 2.1 一次只处理一个系列

哪怕你已经熟悉多个系列，也不要在同一轮任务里混写。因为系列一多，最容易出现的问题就是：

- 标题命名风格漂移
- 开发商 / 发行商记串
- 版本关系混淆
- 目录更新遗漏

所以默认做法应该是：

1. 只接一个系列
2. 只建这个系列目录
3. 只查这个系列资料
4. 只更新与这个系列有关的总目录项

### 2.2 先立目录，再写正文

任何系列任务都应该先建立或更新该系列的 `README.md`，原因很简单：

- 它先决定你这个系列怎么分期
- 它先决定这一批写哪些作品
- 它先决定阅读顺序怎么说明
- 它先把版本关系里的大坑提前标出来

如果没有这个步骤，后面的单作很容易越写越散。

### 2.3 小批次推进，不要一口气写完整个系列

推荐每批只写 2 到 4 作。原因不是偷懒，而是为了控制事实核对质量。

一批作品过多时，最常见的风险有：

- 资料来源开始互相覆盖
- 作品定位写重复
- 某作特殊版本关系被忽略
- 后面几作开始套前面的句式和结论

所以最稳的节奏是：

1. 定本批次范围
2. 写系列 README
3. 写 2 到 4 作
4. 同步更新目录
5. 下一批再继续

### 2.4 每一作都要重新核对，不沿用上一作结论

哪怕是同一系列连续正传，也不能默认：

- 开发商和发行商完全没变
- 发售平台完全一致
- 中文常见译名完全固定
- 加强版 / 重制版 / 导演剪辑版的逻辑一样

系列背景可以共享，但事实不能偷懒复用。

## 3. 标准执行顺序

一轮完整任务，推荐按下面顺序执行。

### 3.1 第一步：查看仓库现状

先做三件事：

1. 看 `Wiki/WIKI_WRITING_GUIDE.md`
2. 看 `Wiki/README.md`
3. 检查该系列目录是否已经存在

如果该系列已经有部分内容，还要补看：

- 现有 `README.md`
- 现有单作文件
- 当前命名方式是否已经固定

这一层的目的不是正式写作，而是先确认：

- 仓库现有风格是什么
- 系列目录有没有既定结构
- 之前是否已经写过某些作品

### 3.2 第二步：确定本批次文件清单

在真正落笔前，应该先明确“这次到底要创建哪些文件”。

标准做法是先列出：

- 系列 `README.md`
- 本批次单作文件
- 需要同步更新的 `Wiki/README.md`

例如：

- `Wiki/某系列/README.md`
- `Wiki/某系列/GAME_A_WIKI.md`
- `Wiki/某系列/GAME_B_WIKI.md`
- `Wiki/README.md`

这样做有两个好处：

- 用户能清楚知道这一批范围
- 写作者自己不会中途失控扩写

### 3.3 第三步：先写系列 README

系列 `README.md` 必须先写，而且建议至少包含：

- 系列概览
- 分期方式
- 阅读顺序
- 当前已覆盖条目
- 版本关系与补充说明
- 后续可补方向

这个文件的作用不是做总百科，而是先回答几个战略性问题：

- 这个系列当前准备怎么拆
- 这批先写哪几作
- 哪些版本最容易混淆
- 读者应该从哪几作开始看

### 3.4 第四步：逐作确认“独特定位”

这是整个流程里很重要、但最容易被跳过的一步。

在写某一作正文前，先用一句话确认它在系列中的独特定位。例如：

- 系列起点作
- 分支叙事最激进的一作
- 开放世界转型作
- 动作化转折点
- 氛围收束作
- 补完性质很强的资料片

这一步的作用，是强迫自己在动笔前先想清楚：

- 这作为什么存在
- 它和前后作最大的差别是什么
- 这一篇正文的重心应该往哪边偏

如果这一步不做，写出来的单作很容易只是“同一篇简介换标题”。

### 3.5 第五步：逐作查资料，逐作落正文

每一作都要独立核对至少这些项目：

- 正式名称
- 常见中文译名
- 开发商 / 发行商
- 发售日期
- 平台
- DLC / 资料片 / 重制版 / 导演剪辑版 / 整合版关系

资料搜集顺序建议固定为：

1. 官方站 / 官方新闻
2. 官方商店页
3. Wikipedia
4. 相关权威 Wiki
5. 最后才做中文归纳

写作时统一采用仓库既定骨架：

1. 标题
2. 开头说明
3. 引言或题记
4. 作品概览
5. 基础信息速览
6. 世界观与背景设定
7. 剧情概要
8. 玩法与结构特色
9. 主要角色与势力
10. 评价与历史定位
11. 一句话总结
12. 参考资料

### 3.6 第六步：同步更新目录

这一轮作品写完后，至少更新两处：

1. 该系列的 `README.md`
2. `Wiki/README.md`

系列目录要更新：

- 当前已覆盖条目
- 本批次新增作品
- 阅读顺序或分期是否需要调整

总目录要更新：

- 增加新的系列入口，或补充该系列已覆盖内容

这一层千万不要拖到“以后再说”。因为一旦单作增加，总目录没跟上，仓库的可用性会立刻下降。

### 3.7 第七步：做一轮收尾自检

最低限度建议检查下面这些项目：

- 文件名是否统一
- 标题是否统一
- 系列内链接是否都能对上
- 总目录入口是否已经加上
- 单作结构是否完整
- 每一作的版本关系是否写清
- 是否把前作信息错写到后作
- 是否引用了明显不可靠来源

如果有时间，再加一层风格自检：

- 有没有写成资料堆
- 有没有写成宣传文案
- 有没有段落明显重复
- 有没有“系列通用套话”过多的问题

## 4. 资料核对的推荐顺序

面对一个新系列，最容易乱的不是写，而是查资料时不知道先信谁。

推荐顺序如下：

### 4.1 先确认官方事实

优先确认这些硬信息：

- 官方正式名称
- 开发商 / 发行商
- 发售平台
- 官方对版本的命名方式

通常优先看：

- 官方站
- 官方新闻
- 官方商店页
- 官方合集 / 完整版说明

### 4.2 再确认版本关系

很多系列真正的坑在这里。要优先分清：

- 原版
- 导演剪辑版
- 加强版
- 重制版
- 高清复刻版
- 整合版
- 合集

尤其要避免：

- 把“高清版”写成“重制版”
- 把“整合版”写成“新作”
- 把“附带模式”写成“主线补完”

### 4.3 最后再做中文整理

中文写作阶段要做的是“整合表达”，不是“把所有资料拼接起来”。

此时才需要处理：

- 中文常见译名怎么写更自然
- 段落怎么组织更顺
- 哪些信息放正文，哪些放速览
- 哪些补充内容只需简写

## 5. 如何防止上下文污染

这是系列化写作最值得单独强调的部分。

### 5.1 不把系列背景等同于单作事实

系列背景可以复用，单作事实不能复用。

例如可以共享：

- 世界观大框架
- 系列主轴冲突
- 常驻人物的大致关系

但不能直接复用：

- 开发商 / 发行商
- 发售日期
- 平台
- 版本关系
- 某作的独特结构

### 5.2 每写一作都先回答一句话

动笔前先回答：

“这作在系列里的独特定位是什么？”

只要这句话答不清，正文大概率也写不清。

### 5.3 同阶段作品可以共享气质，不共享细节

同一阶段的几部作品确实可能气质接近，但仍要逐作确认：

- 哪一作开始动作化
- 哪一作才加入开放世界
- 哪一作的分支最极端
- 哪一作的资料片对主线最重要

不要因为“它们看起来差不多”就偷懒。

### 5.4 一批不要写太多

本质上，小批次就是控制上下文污染的最好办法。

2 到 4 作时，人还能记住每作独特定位；
一口气写 8 到 10 作时，后半段几乎一定会开始混。

## 6. 文件与目录建议

当前项目最推荐的组织方式是：

1. `Wiki/README.md` 作为总入口
2. `Wiki/系列名/README.md` 作为系列入口
3. `Wiki/系列名/单作文件.md` 作为正文页

### 6.1 文件命名建议

默认尽量使用：

- 全大写英文文件名
- 必要时加年份或副标题
- 避免同名冲突

例如：

- `RESIDENT_EVIL_1996_WIKI.md`
- `THE_WITCHER_3_WILD_HUNT_WIKI.md`
- `HITMAN_2016_WIKI.md`

### 6.2 什么时候要带年份

遇到这些情况建议带年份：

- 同名重启
- 原版与重制版并存
- 旧作与新作中文名完全相同

### 6.3 什么时候可以合并写

以下情况可以考虑合并为一篇：

- 本体与极短资料片高度一体
- 几部早期作品体量都较小
- 原版与重制版主要差异集中在版本关系说明

但是否合并，仍要以“不影响事实清晰”为前提。

## 7. 一次任务的推荐检查清单

可以把下面这份表当作最简执行清单：

### 7.1 开始前

- 已阅读 `WIKI_WRITING_GUIDE.md`
- 已查看 `Wiki/README.md`
- 已确认该系列是否已有目录
- 已列出本批次文件清单

### 7.2 写作中

- 已先写系列 `README.md`
- 已确定每一作的独特定位
- 已逐作核对基础事实
- 已区分版本关系
- 已按统一结构写正文

### 7.3 完成后

- 已更新系列目录
- 已更新总目录
- 已检查文件名与链接
- 已检查是否存在系列内事实串用
- 已检查参考资料是否可靠

---

## 8. 推荐的执行节奏

如果把整个流程压缩成最实用的一句话，可以记成：

1. 先看规范和现状
2. 先定本批次文件
3. 先写系列 README
4. 再逐作确认定位并写正文
5. 写完就同步目录
6. 最后统一自检

这个顺序的好处在于，它既能保证信息质量，也能让仓库结构始终保持可浏览、可继续扩写的状态。

---

## 9. 一句话总结

这套工作流的核心，不是“写得更快”，而是“先把边界和顺序定清楚，再把每一作当成独立对象认真核对”，这样系列 Wiki 才能越写越稳，而不是越写越乱。
